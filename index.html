SECTION 2: INTRO NOTES

  • The 4 cornerstones of computational thinking
    1. Decomposition: breaking a problem down into smaller tasks.
    2. Pattern Recognition: find patterns in how the subtasks are solved to streamline/minimize the amount of work done in your code.
    3. Abstraction: filter out unrelated info.
    4. Algorithm: the sequence of steps or rules to solve the problem.
  • Misc. terms
    • Conditional statement = if x meets condition y then x will do a; if x does not meet condition y, then x will do b
    • Debugging = fixing the dang thang
    • For loop = an action that repeats until a given condition is met
    • Terminal and Gitbash are CLIs (command line interfaces)
    • GUI = graphical user interface
    • pair programming = 2 people coding at once
      • driver-navigator model = driver is writing code and nav is directing/overseeing
    • SSH key = access credential for Secure SHell network protocol
    • MVP - minimum viable product. first iteration of a product without any extra features.
  • Misc. info
    • Bash & gitbash are the same thing
    • When stuck on a problem, timebox. Work 20-30 minutes and if still not clicking, ask for help.

SECTION 3: HTML

Notes on HTML
  • The head element contains information about the webpage.
  • The body element represents the visible content shown to the user.
  • The meta element holds metadata (info for the browser, not the user)
  • Parent/child elements are the bigger and littler guys. For example, "p" is the child element to "body", the parent.
  • An element is the entire bit of info from the opening tag to the closing tag. Like this, inside the "li"s!
  • A tag is only what appears inbetween the angle brackets.
  • To make a list, the parent element should be "ul" for an unordered list or "ol" for an ordered list. Every element within should go between "li" tags, for the list items.
  • The "a" tag means "anchor". It sets up links.
  • The href item(?) precedes your url. You can put some words between the link and your closing "/a" to make linked text. Like this!
  • on images...
    1. The item(?) "src" is the source of the image.
    2. The item(?) "alt" is like metadata for the image. You can use it to describe the contents and function of the image so that screen readers and browsers can pick up that info! Great for accessibility and SEO.
  • HTML is fun!!!

SECTION 5: GIT + GITHUB

  • Gitbash commands
    • Non-git Gitbash commands
      • pwd = print working directory
      • ls = list of files and directories in current directory
      • cd [directory] = change directory to the one listed
      • cd .. = move back a directory
      • mkdir [directory] = makes a directory
      • touch [filename].txt = makes file
      • start [filename].txt = opens file (cmd is "open" in terminal)
    • Git Gitbash commands
      • git status = prints current branch
      • git checkout = move from working (current) branch to a new branch
      • git checkout -b [branch name] = creates a new branch and moves you there
      • git add = adds modifications in cwb to staging area
      • git add -A = add all changes
      • git pull [directory] [branch] = recieve a branch's changes into the local environ
      • git push [directory] [branch] = pushes local changes to the remote GH branch. (it can create a new remote branch or add new changes to an existing branch.)
      • origin = source of the pull will be the github repo. (i.e., git pull origin main)
  • GitHub terminology
    • repository/repo = folder for all files in a project
    • branch = the workspace where you can make changes that don't affect the live site
    • markdown (.md) = a file format of plaintext that easily converts into html
    • commit changes = saving a new version of the file you're working on. this includes adding, editing, or deleting files. it adds changes to your branch.
    • pull request (pr) = asking for changes on a branch to merge into another branch. it's a way to discuss and review your work with others.
    • merge = changes are pulled from your branch into the new branch (publishes/deploys into the live site)
    • issue = way to report new tasks or fixes. tracks progress and bug fixes.
    • blocker = the thing keeing you stuck
    • bottleneck = the part slowing down everything else
    • staging area = informs git that the changes there will be added in the next commit.
    • base = branch you're merging to
    • compare = where you're merging from
  • Workflow for content updates in GH
    1. create a branch
    2. add commits
    3. open a pull request
      • title pr descriptively
      • fill body of pr with instructions for team members reviewing it.
        1. changes that were made
        2. how it should function
        3. how to start the application so the feature can be tested
    4. discuss and review
    5. merge and deploy
  • Misc. Git notes
    • When cloning a repo, you need to be inside the directory you want the repo directory to be in!
    • Always do a Git pull before you work to make sure you aren't using old code!
    • Always review with others before merging!
